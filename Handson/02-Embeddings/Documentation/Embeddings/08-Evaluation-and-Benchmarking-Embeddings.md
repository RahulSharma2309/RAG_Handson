# 08 — Evaluation and Benchmarking Embeddings

## Why Evaluation Is Not Optional

Choosing an embedding model without benchmarking is like choosing a search engine by looking at its logo.

Two models with similar MTEB leaderboard scores can produce completely different retrieval quality on your specific corpus with your specific query distribution.

Evaluation is not a one-time task. It is an ongoing discipline:
- before adopting a new model
- before each major re-index
- after any preprocessing change
- when users report retrieval regressions

---

## The Evaluation Foundation: Your Gold Query Set

Everything else depends on having a good gold query set.

A gold query set is a list of (query, expected_relevant_chunks) pairs built from your actual corpus and representing the actual questions your users will ask.

### How to Build One

**Step 1: Collect real or realistic queries**

Use one or more of:
- actual user questions from production logs
- questions written by subject-matter experts
- questions generated by prompting an LLM: "Given this document, write 5 questions a user might ask"

**Step 2: Define expected relevant chunks for each query**

For each query, identify:
- 1-3 chunks that are clearly correct answers
- 1-3 chunks that are partially relevant (acceptable alternates)
- Example chunks that look similar but are wrong (hard negatives)

**Step 3: Cover all difficulty levels**

| Level | Description | Example |
|---|---|---|
| Easy | Exact term overlap between query and chunk | "AWS CLI installation steps" → chunk with AWS CLI instructions |
| Medium | Paraphrase — same meaning, different words | "How to undo a Helm release" → chunk about `helm rollback` |
| Hard | Conceptual — requires domain understanding | "Why would auth fail silently?" → chunk about JWT expiry handling |

Without hard queries, you are measuring vocabulary matching, not semantic retrieval.

**Step 4: Include negative queries**

Queries that should return no highly relevant result (to catch false positives).

## The Core Metrics

### Recall@k

Of the known relevant chunks, what fraction appeared in the top-k retrieved results?

```
Recall@k = relevant chunks found in top-k / total known relevant chunks
```

Example:
- Gold: 3 relevant chunks exist for this query
- Retrieval top-5 contains 2 of them
- Recall@5 = 2/3 = 0.67

Use k = 3, 5, 10 for RAG systems.

Why it matters: if relevant chunks are not retrieved, the LLM cannot answer correctly — regardless of how good the LLM is.

### Precision@k

Of the top-k retrieved chunks, what fraction are actually relevant?

```
Precision@k = relevant chunks in top-k / k
```

Example:
- Retrieved 5 chunks. 3 are relevant.
- Precision@5 = 3/5 = 0.60

Why it matters: too many irrelevant chunks in top-k "dilute" the context passed to the LLM and can cause hallucinations.

### MRR (Mean Reciprocal Rank)

What is the average position of the first relevant chunk across queries?

```
MRR = mean of (1 / rank of first relevant chunk)
```

Example:
- Query 1: first relevant chunk at rank 1 → 1/1 = 1.0
- Query 2: first relevant chunk at rank 3 → 1/3 = 0.33
- Query 3: first relevant chunk at rank 2 → 1/2 = 0.50
- MRR = (1.0 + 0.33 + 0.50) / 3 = 0.61

Why it matters: for single-answer queries, users expect the answer to be the first retrieved result. MRR penalises systems that bury the correct answer.

### nDCG (Normalized Discounted Cumulative Gain)

Rewards relevant results appearing earlier in the ranking.

Discounts results that appear deeper in the list (rank 5 matters less than rank 1).

When to use: when you have graded relevance (perfect/acceptable/poor) rather than binary relevant/not relevant.

### No-Hit Rate

Percentage of queries where the gold relevant chunk did not appear anywhere in top-k.

```
No-hit rate = queries with zero relevant chunks in top-k / total queries
```

A no-hit rate > 5-10% signals a serious retrieval problem.

## Running an Evaluation

### Basic Evaluation Loop

```python
def evaluate(queries, retriever, k=5):
    recall_scores = []
    precision_scores = []
    mrr_scores = []
    no_hits = 0

    for q in queries:
        results = retriever.retrieve(q['query'], k=k)
        retrieved_ids = {r.metadata['chunk_id'] for r in results}
        relevant_ids = set(q['expected_chunk_ids'])

        # Recall
        hits = relevant_ids & retrieved_ids
        recall = len(hits) / len(relevant_ids) if relevant_ids else 0
        recall_scores.append(recall)

        # Precision
        precision = len(hits) / k
        precision_scores.append(precision)

        # MRR
        for rank, r in enumerate(results, start=1):
            if r.metadata['chunk_id'] in relevant_ids:
                mrr_scores.append(1 / rank)
                break
        else:
            mrr_scores.append(0)

        # No-hit
        if not hits:
            no_hits += 1

    return {
        f'Recall@{k}': round(sum(recall_scores) / len(recall_scores), 4),
        f'Precision@{k}': round(sum(precision_scores) / len(precision_scores), 4),
        'MRR': round(sum(mrr_scores) / len(mrr_scores), 4),
        'No-Hit Rate': round(no_hits / len(queries), 4),
    }
```

## Failure Analysis: The Most Valuable Step

Metrics tell you *how much* retrieval fails. Failure analysis tells you *why*.

After every benchmark, manually inspect a sample of failures.

### Failure Category 1: Semantic Miss

The query and the correct chunk mean the same thing but are worded differently.

Example:
- Query: "How do I reverse a Helm deployment?"
- Correct chunk says: "Use `helm rollback` to revert to a previous release."
- Missed because: model does not map "reverse" → "rollback" well.

Diagnosis: embedding model has weak semantic paraphrase capability.  
Fix: try a model with stronger sentence similarity training, or enrich chunk text with alternate phrasings.

### Failure Category 2: Lexical Miss

Query uses exact terms not in the chunk text.

Example:
- Query: "EKS:CreateCluster permission"
- Correct chunk says: "IAM role must have the cluster creation permission."
- Missed because: "EKS:CreateCluster" is an exact identifier; the chunk paraphrases it.

Diagnosis: exact-term retrieval failure.  
Fix: add hybrid retrieval with BM25 for exact token matching.

### Failure Category 3: Truncation Miss

The relevant content was at the tail of a long chunk and was truncated before embedding.

Diagnosis: check `token_count` of the chunk. If near or over operational limit, truncation occurred.  
Fix: reduce chunk size, split the chunk, or apply tail-preserving truncation strategy.

### Failure Category 4: Metadata Over-Filtering

Filter is too restrictive and excludes the correct chunk.

Example: filter `version = "v3"` but the relevant chunk has `version = "v3.0"` (trailing `.0` mismatch).

Diagnosis: metadata value inconsistency.  
Fix: normalize metadata values at ingestion.

### Failure Category 5: Boilerplate Pollution

Retrieved chunks contain repeated headers/footers that make them look similar to each other.

Diagnosis: all retrieved chunks share similar leading text (page headers, confidentiality tags).  
Fix: improve preprocessing to remove boilerplate before embedding.

## Model Comparison Protocol

When comparing two models:

1. Use the same gold query set for all models
2. Use the same chunk artifacts (same preprocessing, same splits)
3. Use the same retrieval configuration (k, metric, filters)
4. Record: Recall@5, Recall@10, Precision@5, MRR, no-hit rate, p95 query latency
5. Record failure categories for each model
6. Compute cost difference at projected monthly volume
7. Decision: model with better Recall@5/MRR + acceptable latency/cost wins

Never compare models using different preprocessing or different chunk profiles — it pollutes the comparison.

## Minimum Acceptable Thresholds (Starting Point)

These are not universal standards — calibrate for your system. But use as a starting point:

| Metric | Minimum Acceptable | Good | Excellent |
|---|---|---|---|
| Recall@5 | 0.65 | 0.80 | 0.90+ |
| MRR | 0.55 | 0.70 | 0.85+ |
| No-Hit Rate | < 0.15 | < 0.08 | < 0.03 |

Below minimum: do not promote to production. Investigate failures first.

## Regression Testing (Ongoing)

Every time something changes in your pipeline, re-run the same benchmark:

Triggers for re-run:
- new embedding model version
- preprocessing rule change
- chunk_size or overlap change
- new documents added to corpus
- metadata schema change

Track metrics over time. A sudden drop in Recall@5 after a change tells you exactly what broke it.
